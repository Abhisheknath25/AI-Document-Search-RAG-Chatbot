AI Document Search (RAG Chatbot)

📋 Section 1 — Project Overview & Architecture
Q1. What is this project and what problem does it solve?

This is an AI Document Search (RAG Chatbot) — a full-stack application that lets users upload PDF documents and ask natural language questions about them. The AI retrieves relevant text chunks and generates accurate answers with inline citations (e.g., [Doc 1], [Doc 2]).

Problem it solves: Regular AI chatbots can't access your private documents and may hallucinate. RAG (Retrieval-Augmented Generation) grounds the AI's responses in your actual uploaded data, eliminating hallucination and providing verifiable, cited answers.

Q2. Explain the RAG (Retrieval-Augmented Generation) architecture used in this project.

RAG is a two-phase architecture:

  1. Retrieval Phase: The user's query is converted into a vector embedding. This embedding is compared against pre-indexed document chunk embeddings using cosine similarity to find the top-K most relevant chunks.
  2. Generation Phase: The retrieved text chunks are injected into the LLM's prompt as context. The LLM then generates a response based only on the provided context, with citations referencing specific source documents and pages.

This ensures the AI's answers are grounded in real data rather than relying solely on its training knowledge.

Q4. How does the end-to-end flow work when a user uploads a PDF and asks a question?
  1. Upload: User drops a PDF → FastAPI receives the file bytes
  2. Extraction: PyPDF extracts text page-by-page
  3. Chunking: RecursiveCharacterTextSplitter splits text into ~1000-character chunks with 200-char overlap
  4. Embedding: Each chunk is converted to a vector using OpenAI/Gemini embedding model
  5. Indexing: Vectors are stored in FAISS (local) or Pinecone (cloud)
  6. Query: User asks a question → query is embedded → top-4 similar chunks are retrieved
  7. Generation: Retrieved chunks + query are sent to GPT-4o-mini/Gemini → answer with citations is returned
  8. Display: React frontend renders the markdown answer with expandable citation cards
Q5. How did you merge the frontend and backend into a single deployment?

The React frontend is built using vite build which outputs static files (HTML, CSS, JS) to the dist/ folder. These files are copied into the FastAPI backend/static/ directory. FastAPI then serves them using:

from fastapi.staticfiles import StaticFiles
app.mount("/", StaticFiles(directory="static", html=True), name="static")

The API URL in the frontend is set to '' (empty string), so all fetch('/api/...') calls are relative to the same origin. This way, a single Python server on port 8000 serves both the API and the UI.

🐍 Section 2 — Backend (Python / FastAPI)
Q6. Why did you choose FastAPI over Flask or Django?
  • Async support: FastAPI is built on ASGI with native async/await, essential for non-blocking file uploads and API calls to LLM providers.
  • Automatic docs: Generates Swagger UI at /docs and ReDoc at /redoc automatically.
  • Pydantic validation: Request/response models are validated at the framework level using type hints.
  • Performance: One of the fastest Python frameworks, comparable to Node.js and Go.
  • Lightweight: No ORM, admin panel, or template engine bloat — perfect for an API-only backend.

Q8. How does the PDF processing pipeline work?

The PDFProcessor class handles two stages:

  1. Text Extraction: Uses PyPDF.PdfReader to read the uploaded bytes, iterates through each page, extracts text, and creates Document objects with metadata (source filename, page number).
  2. Chunking: Uses LangChain's RecursiveCharacterTextSplitter with:
    • chunk_size=1000 characters
    • chunk_overlap=200 characters (ensures context isn't lost at chunk boundaries)
    • Separators: ["\n\n", "\n", " ", ""] (splits at paragraphs first, then lines, then words)

The metadata (source, page) propagates through chunking so citations remain accurate.

Q9. How do you handle API keys securely? They're passed in headers — is that safe?

API keys are passed via custom HTTP headers (x-api-key, x-pinecone-key). This is a valid approach because:

  • Headers are encrypted in transit when using HTTPS.
  • They are not logged in URL access logs (unlike query parameters).
  • Keys are stored in the browser's localStorage on the client side — they never touch the backend's filesystem or database.

For production improvements: Use environment variables on the server, implement OAuth2 or JWT-based auth, and use a secrets manager like AWS Secrets Manager or HashiCorp Vault.

Q10. What is CORS and why is it configured in this project?

CORS (Cross-Origin Resource Sharing) is a browser security mechanism that blocks requests from one origin (e.g., localhost:5173) to another (e.g., localhost:8000).

During development, the React frontend runs on port 5173 and the FastAPI backend on port 8000 — these are different origins. Without CORS middleware, all API calls from the frontend would be blocked by the browser.

app.add_middleware(
    CORSMiddleware,
    allow_origins=["*"],  # Allow all origins (development only)
    allow_methods=["*"],
    allow_headers=["*"],
)

In production: Replace "*" with specific allowed origins like ["https://yourdomain.com"].

🧠 Section 3 — LangChain, Embeddings & LLM
Q11. What is LangChain and why is it used here?

LangChain is a Python framework for building applications powered by LLMs. In this project, it provides:

  • Document abstraction: The Document class that stores text + metadata (source, page).
  • Text splitting: RecursiveCharacterTextSplitter for intelligent chunking.
  • Embedding wrappers: Unified API for OpenAI and Google embedding models.
  • LLM wrappers: ChatOpenAI and ChatGoogleGenerativeAI with consistent interfaces.
  • Message types: SystemMessage and HumanMessage for prompt construction.
  • FAISS integration: langchain-community provides the FAISS vector store wrapper.
Q12. What are embeddings and how are they used in this project?

Embeddings are dense numerical vector representations of text that capture semantic meaning. Similar texts have similar vectors (high cosine similarity).

In this project:

  1. Each document chunk is converted to an embedding vector (e.g., 1536 dimensions for OpenAI's text-embedding-3-small).
  2. These vectors are stored in the vector database.
  3. When a user asks a question, the query is also embedded into the same vector space.
  4. The system finds the top-K chunks whose vectors are closest to the query vector (cosine similarity).

This is fundamentally how semantic search works — it understands meaning, not just keywords.

Q13. How is the LLM prompt engineered in this project?

The prompt uses a System + Human message pattern:

  • SystemMessage: Contains the RAG instructions — answer only from context, cite sources as [Doc 1], use markdown formatting, and admit when information is not available.
  • HumanMessage: Contains the user's actual query.

The context blocks are formatted with labels like [Doc 1] Source: report.pdf (Page 3) so the LLM can reference them in citations. The temperature=0.2 keeps responses factual and deterministic.

Q14. Why does the project support both OpenAI and Google Gemini? How is switching implemented?

Why: Provides flexibility — users can choose based on cost, availability, or preference. OpenAI is industry-standard but paid; Gemini offers a generous free tier.

How: The provider parameter (passed via the x-provider header) controls which models are instantiated. Both _get_llm() and _get_embeddings_model() use a simple if/else to instantiate the correct LangChain wrapper. The interface is identical thanks to LangChain's unified abstraction — both return objects with the same .invoke() and .embed_documents() methods.

Q15. What is chunk overlap and why is it set to 200?

Chunk overlap means adjacent chunks share some text at their boundaries. With chunk_size=1000 and chunk_overlap=200:

  • Chunk 1: characters 0–1000
  • Chunk 2: characters 800–1800 (overlaps 200 chars with chunk 1)
  • Chunk 3: characters 1600–2600 (overlaps 200 chars with chunk 2)

Why 200: If a key sentence falls exactly at a chunk boundary, without overlap it would be split across two chunks and neither chunk alone would have the full context. The 200-char overlap ensures important sentences near boundaries appear in at least one complete chunk. 200 is roughly 1-2 sentences — enough to preserve context without creating excessive duplication.

🗄️ Section 4 — Vector Store & Search
Q16. What is FAISS and how is it used in this project?

FAISS (Facebook AI Similarity Search) is a library by Meta for efficient similarity search over dense vectors. In this project:

  • Document chunk embeddings are indexed using FAISS.from_documents().
  • The index is saved locally to disk using save_local() and loaded with load_local().
  • Queries use similarity_search_with_score() which returns the top-K matching documents with their L2 distance scores.

FAISS is optimized for speed — it can search millions of vectors in milliseconds using techniques like product quantization and inverted file indexes.

Q17. Why does the project have a NumPy fallback vector store?

FAISS requires C++ compilation and platform-specific binaries (faiss-cpu). It can fail to install on certain systems (e.g., some Windows configs, ARM architectures). The SimpleNumPyVectorStore is a pure-Python fallback that:

  • Uses numpy.dot() for cosine similarity calculations
  • Stores documents and embeddings in Python lists
  • Serializes to disk using pickle
  • Works on any system that has NumPy installed

The project uses a try/except on import — if FAISS isn't available, it falls back to the NumPy store transparently.

Q18. Explain cosine similarity and how it's implemented in the NumPy store.

Cosine similarity measures the angle between two vectors (0 = perpendicular/unrelated, 1 = identical direction/meaning).

Formula: cos(θ) = (A · B) / (||A|| × ||B||)

Implementation in the project:

q = np.array(query_embedding)
matrix = np.array(self.embeddings)
scores = np.dot(matrix, q) / (matrix_norm * q_norm)
top_k_indices = np.argsort(scores)[::-1][:k]

It computes the dot product of the query vector against all stored vectors, normalizes by their magnitudes, then sorts to get the top-K highest-scoring matches.

Q20. How does document deletion work in the vector store?

The delete_document() method filters by the source metadata field:

  1. Iterates through all stored documents and embeddings in the NumPy store
  2. Keeps only those whose metadata["source"] does NOT match the filename to delete
  3. Saves the filtered list back to disk
  4. If FAISS index exists, deletes the entire FAISS directory (since FAISS doesn't support metadata-based deletion natively) — it rebuilds lazily on the next query/upload
⚛️ Section 5 — Frontend (React / TypeScript)
Q21. Describe the frontend architecture and component structure.

The frontend follows a container + presentational component pattern:

  • App.tsx (Container): Holds all state (config, documents, messages), handles API calls, and passes data down as props.
  • Sidebar.tsx (Presentational): Displays configuration controls (provider toggle, API key, vector storage, PDF upload zone, document list).
  • ChatArea.tsx (Presentational): Renders the chat interface — messages, markdown rendering, citation cards, and the input form.

State flows top-down via props; events flow bottom-up via callback functions.

Q22. How does the drag-and-drop PDF upload work?

The upload zone uses the HTML5 Drag and Drop API:

  1. onDragEnter / onDragOver → Sets dragActive=true to highlight the drop zone
  2. onDragLeave → Resets the highlight
  3. onDrop → Captures e.dataTransfer.files, calls onUpload(files)
  4. The upload function creates a FormData, appends files, and sends a POST /api/upload with API config headers
  5. A hidden <input type="file"> via useRef is also available for click-to-browse

The zone visually responds with a purple border highlight during drag, and shows a spinner with "Processing PDF... Chunking & embedding" during upload.

Q23. How is the chat state managed? Why not use Redux or Zustand?

All state is managed using React's built-in useState hooks in the App component:

  • messages: Array of {role, content, citations} objects
  • isLoadingChat: Boolean to show the typing indicator
  • Config states (provider, apiKey, etc.) are persisted to localStorage

Why no Redux/Zustand: The app has a simple, tree-shaped data flow with only ~10 state variables and 3 components. Adding a state management library would be over-engineering — React's native hooks handle this elegantly with zero boilerplate.

Q24. What is the glassmorphism design pattern used in the UI?

Glassmorphism is a modern UI design trend that creates a frosted glass effect using:

  • backdrop-filter: blur(16px) — blurs the background behind the element
  • Semi-transparent backgrounds: rgba(18, 20, 29, 0.55)
  • Subtle borders: border: 1px solid rgba(255, 255, 255, 0.05)
  • Layered radial gradients on the body for depth

The project defines reusable CSS classes: .glass, .glass-panel, .glass-card, and .glass-input for consistent application across components.

Q25. How does the markdown rendering and citation system work in the chat?

AI responses are rendered using the react-markdown library, which converts markdown (bold, lists, code blocks, headings) into styled HTML inside a .prose container with custom CSS.

Citations are handled as a separate expandable section below each AI message:

  • The backend returns sources array with {id, document, page, text} for each chunk
  • A collapsible panel shows "Source Citations (N)" with a toggle button
  • Each citation card displays the document name, page number, and the exact text excerpt used
  • State is tracked via expandedCitations — a Record mapping message index to boolean
🐳 Section 6 — Deployment & DevOps
Q26. Explain the Docker multi-stage build in this project.

The Dockerfile uses two stages to minimize the final image size:

  1. Stage 1 — Frontend Builder (node:20-alpine): Installs npm dependencies, runs npm run build, produces the dist/ output. This stage is discarded after the build.
  2. Stage 2 — Backend Runtime (python:3.10-slim): Installs Python dependencies, copies backend code, then copies the built frontend from Stage 1 into ./static.

Result: The final image contains only the Python runtime + compiled frontend assets — no Node.js, no node_modules, no source TypeScript files.

Q27. How would you deploy this to production?
  • Cloud Platforms: Deploy the Docker container to AWS ECS/Fargate, Google Cloud Run, Azure Container Instances, or Railway/Render.
  • Environment: Set API keys via environment variables (not headers) using a secrets manager.
  • HTTPS: Use a reverse proxy (Nginx/Caddy) or cloud load balancer for SSL termination.
  • CORS: Restrict allow_origins to your specific domain.
  • Vector Store: Use Pinecone (cloud) instead of FAISS (local) for persistence across container restarts.
  • Scaling: Use horizontal scaling with a shared vector store; ensure the app is stateless.
Q28. What testing strategy is used in this project?

The project uses Python's unittest framework with unittest.mock for mocking:

  • test_pdf_processing_chunks: Verifies that the text splitter produces chunks with correct metadata.
  • test_simple_numpy_vector_store: Tests the NumPy fallback store — adds documents with mock embeddings, queries, and verifies the correct document is returned with high similarity.
  • test_vector_manager_add_and_search: Mocks the embeddings model using @patch, tests end-to-end add + search + list operations.

Mock embeddings avoid real API calls, making tests fast, free, and deterministic.

Q30. If you had to scale this to handle 1 million documents, what changes would you make?
  • Vector Store: Replace FAISS with Pinecone, Weaviate, or Qdrant (cloud-managed, distributed).
  • Async Processing: Use a task queue (Celery + Redis) for PDF processing — uploads return immediately, processing happens in background workers.
  • Chunking: Pre-compute and cache embeddings; use batch embedding APIs instead of per-document calls.
  • Database: Add PostgreSQL for document metadata (user ownership, upload dates, tags).
  • Caching: Cache frequent queries and their results using Redis.
  • Load Balancing: Run multiple FastAPI instances behind an Nginx load balancer or use Kubernetes.
  • Observability: Add logging (ELK stack), metrics (Prometheus/Grafana), and tracing (OpenTelemetry) for monitoring.
